# Rebuilding `wdosbox-x.wasm` with General MIDI (FluidSynth)

`Web/emulators/wdosbox-x.wasm` is a **custom build** of js-dos's DOSBox-X
emulator with the **FluidSynth** software synthesizer compiled in, so games that
use General MIDI / MPU-401 (most LucasArts & Sierra adventures, many RPGs) get
real SoundFont music instead of silence. The stock js-dos `wdosbox-x.wasm` ships
with the MIDI *dispatcher* but **no synth linked**, so `mididevice=fluidsynth`
falls back to a null handler and music is silent.

This directory documents how that binary is produced and keeps the source
modifications (GPL-2 / LGPL-2.1 corresponding source).

## Why a rebuild was needed

DOSBox-X bundles the full FluidSynth sources under
`src/libs/fluidsynth/`, but they only compile on Windows (`#if !C_FLUIDSYNTH &&
defined(WIN32)`); with `C_FLUIDSYNTH` defined the code expects an *external*
system `libfluidsynth`, which doesn't exist for WebAssembly. The patches make
the bundled FluidSynth compile under Emscripten and link statically, then enable
the handler with `C_FLUIDSYNTH=1`.

## Toolchain

- **Emscripten 6.0.0** (`emcc` + its bundled Binaryen 130).
- **CMake** + **Ninja** (Homebrew).

The js-dos Dockerfile pins Emscripten **3.1.28**, but that does NOT work for
this larger (FluidSynth-enabled) module:
- 3.1.28's bundled **Binaryen 111 crashes** on the Asyncify pass
  (`UNREACHABLE … Asyncify.cpp:1142, "unexpected expression type"`).
- Swapping in a newer standalone Binaryen 130 fixes the crash, but then the
  `.wasm` (minified by 130) and the `.js` (generated by emcc 3.1.28) disagree on
  the minified wasm import names → `import a:a must be callable` at instantiate
  (black screen), and 3.1.28 has no supported way to disable that minification.

Building everything with **Emscripten 6.0.0** avoids both: its JS generator and
its bundled Binaryen 130 are matched, so import-name minification is self-
consistent. 6.0.0 also emits the `wasmImports` JS style the repo's vendored
js-dos framework already uses.

## Reproduce

```sh
# 1. Source tree (js-dos emulators + DOSBox-X submodule)
git clone --depth 1 https://github.com/caiiiycuk/emulators.git emulators-src
cd emulators-src
git config url."https://github.com/".insteadOf "git@github.com:"
git submodule update --init -- native/dosbox native/dosbox-x native/sokol-lib native/sokol-tools-bin

# 2. Apply PocketDOS FluidSynth patches
git -C native/dosbox-x apply ../../PocketDos/tools/midi-build/patches/dosbox-x-fluidsynth.patch
git apply PocketDos/tools/midi-build/patches/emulators-jsdos.patch

# 3. Toolchain — Emscripten 6.0.0 (bundled Binaryen 130; see Toolchain note for
#    why 3.1.28 from the js-dos Dockerfile does NOT work)
git clone https://github.com/emscripten-core/emsdk.git ../emsdk
(cd ../emsdk && ./emsdk install latest && ./emsdk activate latest)   # 6.0.0 here
brew install cmake ninja

# 4. Build
source ../emsdk/emsdk_env.sh
export PATH="$(brew --prefix)/bin:$PATH"
emcmake cmake -GNinja -DCMAKE_BUILD_TYPE=Release -S . -B build/wasm
ninja -C build/wasm wdosbox-x

# 5. Add the worker-server footer to the loader (what gulp's `emulators` task does)
cat build/wasm/wdosbox-x.js src/dos/dosbox/ts/worker-server.js \
  | sed 's/@MODULE_NAME@/WDOSBOXX/g' > wdosbox-x.js

# 6. Install into the app
cp wdosbox-x.js                       ../PocketDos/Web/emulators/wdosbox-x.js
cp build/wasm/wdosbox-x.wasm          ../PocketDos/Web/emulators/wdosbox-x.wasm
cp build/wasm/wdosbox-x.js.symbols    ../PocketDos/Web/emulators/wdosbox-x.js.symbols
```

## What the patches do

`patches/dosbox-x-fluidsynth.patch` (DOSBox-X submodule):
- `include/fluidsynth/config_emscripten.h` (new) — POSIX/WASM FluidSynth config
  (no `<io.h>`/`<windows.h>`, no conflicting `socklen_t`, `HAVE_PTHREAD_H`,
  socket headers for the unused-but-compiled server helper, `TRUE`/`FALSE`/
  `boolean`).
- `include/fluidsynth/config_win32.h` — redirect to the Emscripten config under
  `__EMSCRIPTEN__` (covers files that include the win32 config directly).
- `src/libs/fluidsynth/*.c` — widen the `!C_FLUIDSYNTH && WIN32` whole-file
  guards to also fire on `__EMSCRIPTEN__` so the bundled synth compiles
  (`fluid_dll.c` left out — it's the Windows DLL loader, unneeded statically).
- `fluidsynth_priv.h` — include the Emscripten config under `__EMSCRIPTEN__`.
- `fluid_atomic.h` — use the GCC `__atomic` builtins path on Emscripten.
- `fluid_sys.h` — `FLUID_INLINE` → `static inline` on Emscripten (so `-Oz`
  emits out-of-line copies; otherwise `get_timespec` is undefined at link).
- `src/hardware/imfc.cpp` — explicit out-of-line default ctors for
  `VoiceDefinition` / `InstrumentParameters` / `YmChannelData` (clang `-Oz`
  otherwise drops the implicit `linkonce_odr` copies → undefined at link).

`patches/emulators-jsdos.patch` (emulators repo):
- `targets/dosbox-x-sdl2.cmake` — `-DC_FLUIDSYNTH=1` on `libdosbox-x-jsdos`;
  add the stub source to the `wdosbox-x` target; rename the removed
  `-sASYNCIFY_WHITELIST` flag to `-sASYNCIFY_ONLY` (Emscripten 6.0).
- `CMakeLists.txt` — demote clang-16+ default-error diagnostics
  (`incompatible-pointer-types`, `int-conversion`, etc.) back to warnings; the
  bundled DOSBox-X libs (speexdsp) trip them and the project's `-w` doesn't
  suppress default-errors.
- `native/jsdos/jsdos-emscripten-stubs.c` (new) — stub for
  `pthread_attr_setschedpolicy` (declared but unimplemented in Emscripten's
  single-threaded libc; referenced by FluidSynth's unused thread path).

## SoundFont

`Web/soundfont.zip` contains `TIMGM6MB.SF2` (TimGM6mb, **GPL-2**, license-
compatible, ~5.7 MB). The harness injects it into a game's `C:\` only when the
game's config requests `mididevice=fluidsynth` (see `Web/index.html`
`resolveBundleUrl`), so the memory cost is paid only by GM-enabled games.
Source: Debian `timgm6mb-soundfont` package.
