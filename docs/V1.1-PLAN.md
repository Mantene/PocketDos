# PocketDOS v1.1 — "Harden the wizard + make Win98 pleasant to run"

Scope decided 2026-07-08, after v1.0 was submitted to App Review. v1.1 deliberately
does **not** add new capability — it hardens the one path that already works (the
Win98 SE install wizard) and reduces the per-boot cost of running a Win98 machine.

**Why this and not features:** the wizard succeeded exactly once on device (run #7),
on one device with one ISO. The moment v1.0 is live, real users stress it across an
untested device/ISO matrix, and it *will* fail in new ways. v1.1's job is to make
those failures clean and invisible, and to make the happy path faster — before we
multiply the untested surface with new editions or a new form factor.

**Explicitly deferred to v1.2** (see [Non-goals](#non-goals)): Windows 98 FE / 95 /
ME editions, iPad support.

verification pass, not two. Items #1 and #4 are fully sim/test-verifiable first.

## 1. Failed-install cleanup — ✅ DONE

**The bug (quoted cause):** `InstallOrchestrator.run()` deliberately keeps the install
folder on failure (`Sources/InstallOrchestrator.swift` — "the only forensic artifact
of a long unattended run"), while `GameStore.loadGame()` only recognizes a folder as a
game if it has `drive/sockdrive.metaj` or a `.jsdos`/`.zip` bundle. A failed install
has neither, so `reload()` silently dropped it (`return nil`) → an invisible
`Documents/Games/<uuid>/` folder leaking ~150 MB that the user could never see.

**Fix (at the seam, forensics preserved — no auto-delete):**
- `GameStore.reload()` now collects the folders `loadGame` rejects into a new
  `@Published var orphanedInstalls: [OrphanedInstall]` (id, url, sizeBytes), biggest
  first. `folderSize` sums logical file sizes via metadata enumeration only.
- `orphanedInstallsTotalBytes` + `cleanupOrphanedInstalls()` (deletes every orphan
  folder, refreshes; games are never in the set, so a real game can't be swept up).
- `LibraryView` shows a reclaim banner ("N unfinished installs · Using X MB · your
  games aren't affected" → "Clean Up" → confirmation dialog). The library body was
  restructured so the banner shows even when there are **no games yet** (first install
  failed) — otherwise the user saw "No games yet" with the space still stuck.

**Verification:** 4 new `GameStoreTests` (detection, the "valid games are never
orphaned" safety guard, exact size 100,016 B, delete-only-orphans-keep-games) — all
green against a real temp directory. Full suite **222/222**. Two clean device-sim
screenshots confirmed the two restructured render states (only-orphans; games+orphans)
— see `~/Downloads/pdos-v11-verify/`. The live button tap wasn't observed (a shared,
contended simulator kept stealing the foreground), but the delete is unit-proven and
the button→dialog binding is boilerplate over that method.

## 2. Welcome-tour / first-boot polish — ⬜

The freshly installed Win98 desktop shows the "Welcome to Windows 98" tour on first
boot. It's the first thing the user sees, and it's a mild "this looks unfinished" cue.

**Design fork (decide when starting):**
- **(A) Overlay registry edit** — set `Software\Microsoft\Windows\CurrentVersion\Run`
  to drop the `welcome.exe` autorun, applied on-device via `FAT32OverlayEditor` (same
  machinery as the mouse fix). Deterministic, no runtime keystrokes. Preferred.
- **(B) Scripted first-boot dismiss** — send a close keystroke after the desktop
  settles. Simpler, but timing-dependent and racy (the exact failure class we fought
  in the stage loop).

Recommendation: **(A)** — it's a disk-state fact, not a timing gamble, and reuses a
proven code path.

## 3. stage2→3 flat-exit — ⬜ (device-only verification)

**Today:** the wizard bridges Setup's stage 2 → stage 3 by burning a 30-minute idle
deadline (the "30-min idle toll"). Every successful install wastes ~30 min + battery
waiting for a timeout that we could detect directly.

**Spec (from prior analysis):** after the phase-2 script has capped its cycles **and**
6 consecutive flat capture ticks, hand off to stage 3 **without drawing the death
budget**. The capture stream already tells us the pages are answered and nothing is
growing; that's the same signal the 30-min timer is a crude proxy for.

**Risk:** touches the stage machine that took 7 device runs to stabilize. TDD the
pure logic in `InstallFlow` (a `Stage2FlatComplete` detector, unit-tested like
`RegressedRunDetector`/`CapturePlateauDetector`), keep the 30-min deadline as the
fallback, and verify on device. A prior agent attempt died on an API outage with the
tree left clean — re-dispatch from this spec.

## 4. Overlay consolidation — ⬜ (design first; highest leverage)

**Why it's the sleeper win:** every Win98 boot reseeds the entire write overlay
(~154 MB today) from IndexedDB into the in-memory store before the guest runs — the
same churn behind the stage-2 OOM roulette. Flattening the overlay back into the
immutable chunk store shrinks that reseed toward zero, so it's felt on **every**
launch, not just installs. `FAT32OverlayEditor` already has the composite-read
machinery (overlay ∥ chunks ∥ zeros) needed to fold the overlay into chunks.

**The real design decisions (resolve before coding):**
1. **When to flatten** — on idle after boot? once after a successful install? on app
   background/quit? Each trades responsiveness vs. safety.
2. **Crash-safety mid-consolidation** — writing flattened chunks while the overlay is
   still authoritative must never leave a torn state. Likely: write new chunks + new
   `metaj` to a temp dir, validate (mtools oracle, as inc 4 did), then atomically swap
   and truncate the overlay — never mutate the live chunk set in place.
3. **Interaction with sockdrive identity** — consolidation changes the chunk set, so
   the metaj content-hash (iCloud save identity) changes. Must preserve or re-map the
   game's `contentHash` so saves don't orphan.

Recommendation: a background `OverlayConsolidator` engine (Foundation-only, TDD'd
against the mtools oracle like the other engines), triggered **once after a successful
install completes** for v1.1 (simplest safe trigger), with idle-time consolidation of
imported games deferred.

## Non-goals (v1.2+)

- **Editions** (effort order): 98 FE (cheap — same `\WIN98`/MSBATCH/key format, verify
  the runbook) → 95 OSR2 (own page script, 10-digit key UI, `\WIN95`; retail 95 CDs
  aren't bootable) → ME (ignores MSBATCH → full keystroke phase-1, RAM-tight). When
  starting #2, refactor to an `EditionProfile { dirName, keyFormat, msbatch template,
  script table, SYSTEM.INI lines, floppy quirks }`; the dboxmpi/mouse fix, stage loop,
  and all engines are edition-agnostic.
- **iPad** — mostly layout + 13" screenshots; no emulator-engine risk.
- **iCloud sync of sockdrive diffs**, full MFi controller support.
